# 通用全栈开发规范

> 适用于任何 Web 全栈项目，不绑定特定技术栈。

---

## 一、开发流程

### 1.1 核心原则：小步迭代，即时验证

```
实现一个功能 → 编写测试 → 运行测试 → 修复问题 → 下一个功能
```

**为什么这样做：**
- 问题发现越早，修复成本越低
- 积累多个功能后再测试，问题难以定位
- 测试通过 = 功能完成，可以放心继续

**反模式：**
```
❌ 实现所有功能 → 最后一起测试 → 一堆错误 → 花几小时调试
```

### 1.2 接口开发流程

```
1. 设计接口（URL、参数、返回值）
2. 实现后端接口
3. 编写单元测试
4. 运行测试，确认通过
5. 前端调用接口
6. 检查网络请求和响应
7. 确认功能正常
8. 提交代码
```

### 1.3 数据库变更流程

```
1. 修改数据模型
2. 验证表结构是否正确
3. 测试增删改查
4. 检查数据隔离（如果有用户系统）
```

## 二、测试规范

### 2.1 测试时机

- **每完成一个函数/方法** → 写单元测试
- **每完成一个接口** → 写接口测试
- **每完成一个页面** → 手动测试交互

### 2.2 测试覆盖

至少覆盖：
- 正常流程（Happy Path）
- 参数错误
- 资源不存在
- 权限不足（如果有权限系统）

### 2.3 测试失败处理

```
测试失败 → 看错误信息 → 定位问题 → 修复 → 重新运行测试
```

不要跳过失败的测试，不要注释掉测试。

## 三、代码规范

### 3.1 异常处理

```java
// ✅ 自定义异常，语义清晰
throw new BusinessException(ErrorCode.USER_NOT_FOUND, "用户不存在");

// ❌ 通用异常，丢失上下文
throw new RuntimeException("出错了");
```

### 3.2 命名规范

- 类名：名词（`UserService`、`OrderController`）
- 方法名：动词（`createUser`、`findByEmail`）
- 常量：全大写（`MAX_RETRY_COUNT`）
- 变量：驼峰（`userName`、`orderList`）

### 3.3 注释规范

```java
// ✅ 解释为什么这样做
// 使用 BCrypt 因为它自动加盐，安全性更高
String encoded = passwordEncoder.encode(rawPassword);

// ❌ 解释代码在做什么（代码本身已经说明）
// 加密密码
String encoded = passwordEncoder.encode(rawPassword);
```

## 四、安全规范

### 4.1 密码安全

- 永远不要存储明文密码
- 使用单向哈希算法（BCrypt、Argon2）
- 不要在日志中打印密码

### 4.2 敏感信息

- API Key、数据库密码等放在环境变量或配置文件
- 不要硬编码在代码中
- 不要提交到 Git（加入 .gitignore）

### 4.3 接口安全

- 登录状态用 Session 或 JWT
- 敏感操作需要验证身份
- 用户数据必须按用户隔离

## 五、调试规范

### 5.1 后端调试

```
1. 看控制台日志（错误堆栈）
2. 在关键位置加日志（入参、出参）
3. 用断点调试（IDEA/VS Code）
4. 检查数据库数据
```

### 5.2 前端调试

```
1. 浏览器 F12 → Console（看错误）
2. 浏览器 F12 → Network（看请求/响应）
3. 浏览器 F12 → Application（看 Cookie/Storage）
4. 加 console.log 追踪流程
```

### 5.3 前后端联调

```
1. 先用 Postman/curl 测试后端接口
2. 确认后端正常后，再让前端调用
3. 如果前端报错，先检查 Network 请求是否正确
4. 再检查后端日志
```

## 六、Git 规范

### 6.1 提交频率

- 每完成一个功能提交一次
- 不要积攒大量改动后一次性提交
- 提交前先 `git diff` 检查改动

### 6.2 Commit Message

```
<type>: <description>

feat: 添加用户注册功能
fix: 修复登录状态丢失问题
docs: 更新 API 文档
refactor: 重构订单查询逻辑
```

### 6.3 分支管理

```
main        - 生产代码
develop     - 开发代码
feature/*   - 功能分支
fix/*       - 修复分支
```

## 七、环境管理

### 7.1 配置分离

```
开发环境：.env.development
测试环境：.env.test
生产环境：.env.production
```

### 7.2 配置内容

```
数据库连接
Redis 连接
API Key
第三方服务配置
```

### 7.3 配置原则

- 不同环境使用不同配置
- 敏感信息不提交到 Git
- 使用环境变量或配置文件

## 八、代码审查清单

### 提交前检查

- [ ] 代码能正常运行
- [ ] 测试全部通过
- [ ] 没有硬编码的敏感信息
- [ ] 没有遗留的调试代码（console.log、System.out.println）
- [ ] 命名清晰，代码可读
- [ ] 异常处理完善

### 功能完成检查

- [ ] 接口文档已更新
- [ ] 前后端联调正常
- [ ] 边界情况已处理
- [ ] 错误提示用户友好

## 九、常见错误避免

### 9.1 后端常见错误

```
❌ 直接返回数据库实体给前端
✅ 返回 DTO，隐藏敏感字段

❌ 在循环中查询数据库
✅ 批量查询，内存处理

❌ 忽略异常
✅ 捕获异常，记录日志，返回友好提示
```

### 9.2 前端常见错误

```
❌ 硬编码后端地址
✅ 使用环境变量或代理

❌ 不处理 loading 和 error 状态
✅ 显示加载中和错误提示

❌ 直接修改 state
✅ 使用 setState 或 dispatch
```

## 十一、问题解决流程

```
1. 明确问题是什么（错误信息、复现步骤）
2. 定位问题在哪里（前端/后端/数据库/网络）
3. 分析原因（看日志、调试、搜索）
4. 验证修复（运行测试、手动测试）
5. 记录解决方案（避免下次再犯）
```

---

## 十二、性能考虑

### 数据库

- 避免 N+1 查询
- 合理使用索引
- 大数据量分页查询

### 前端

- 避免不必要的重渲染
- 大列表使用虚拟滚动
- 图片和资源压缩

### 接口

- 设置合理的超时时间
- 大文件异步处理
- 频繁调用加缓存
